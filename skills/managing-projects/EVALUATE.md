# How to evaluate projects

## Concept

A project is obliged to have:

1. **A clear goal** — what are you building?
2. **A clear "good enough"** — what satisfies v1.0?
3. **A clear ending** — when does this stop?

## Evaluate existing projects workflow

Copy this checklist and track your progress:

```text
Evaluate project progress:
- [ ] Step 1: Ask if the user wants to evaluate all projects or a specific one.
- [ ] Step 2: Fetch project info acccordingly
- [ ] Step 3: Score project info
- [ ] Step 4: Give verdicts
- [ ] Step 5: Check the tasks
- [ ] Step 6: Check deadlines and push for completion or deletion
- [ ] Step 7: Verify alignment
```

## Workflow

### Step 1: Ask if the user want to evaluate all project or a specific one

### Step 2: Fetch project info accordingly

1. Fetch projects with `list-projects`. For targeted review, fetch tasks with `list-tasks`
   Push this concept: You don't need more ideas or better planning. You need to kill projects. Publicly. Permanently. A project you're not actively working on is not "waiting" — it's dead. Call it dead. The emotional resistance you feel reading this sentence is the exact thing to pay attention to.

### Step 3: Score project info

Score each project:

| Criterion          | Healthy                                      | Unhealthy                                    |
| ------------------ | -------------------------------------------- | -------------------------------------------- |
| Clear goal         | Specific, measurable purpose/desired_outcome | Vague or missing                             |
| Name as outcome    | Describes the end result/product             | Describes an activity or is vague            |
| Definition of done | Binary checkable requirements name or desc.  | Open-ended or missing                        |
| All fields filled  | name, description, purpose, deadline all set | Any field empty                              |
| Time constraint    | Realistic deadline set                       | No deadline or expired                       |
| Scope discipline   | Focused tasks aligned to done list           | Task bloat or unrelated tasks                |
| Status accuracy    | Status matches actual progress               | Stale (e.g., "In Progress" with no activity) |
| Language           | Everything in English                        | Mixed languages or non-English               |

### Step 4: Give verdicts

- Verdict per project: **Healthy**, **Needs Attention**, or **At Risk**
- Specific recommendation: define done list, set deadline, cut tasks, update status, or close

### Step 5 - Check the tasks

Mid-Project Check (The Middle)

The middle is where most projects die or stall. Ask:

1. Do these features still make sense?
2. Do these requirements still make sense?
3. What can I remove?

Rules:

- **CUT CUT CUT**: The user is a maniac hoarder. Evaluate if the tasks make sense to the end goal and if not force him to delete.
- **No redesigns**
- **No new features** (write them down as v2.0)
- It's OK to cut things out
- It's OK to add more time IF you truly commit and check in again
- Sometimes the right answer is to **quit** — that's not failure

### Step 6: Check deadlines and push for completion or deletion

- For projects that have due dates near with a bunch of tasks, or passed due dates push the user to eliminate projects he could not finish or did not touched
  - It is OK to cut and OK to quit

### Step 7: Check for alignment

- The user has a chronic tendency to spend time on things that are not his main focus or goals. Check active if active projects align with his life goals related in `📋Planning/*`
- Be hard and harsh if needed here. He needs to learn to do less and focus only on what matters.
